<Title>Styling</Title>

`JBCheckbox` styling is theme-driven. Use common design tokens for shared product language, component variables for the checkbox API, and CSS parts only for exposed surfaces that need more targeted styling.

## Common Design Tokens

Use the core theme docs as the source of truth for shared design-system variables:

- [Theme Colors](../../jb-core/theme/stories/colors.mdx) explains the general color palette variables, default colors, customization approaches, and color generation helpers.
- [Theme Sizes](../../jb-core/theme/stories/sizes.mdx) explains shared size and radius variables.
- [Theme Custom Galleries](https://javadbat.github.io/design-system/?path=/docs/theme-custom-material--docs) gathers the shared custom style recipes across components.

Checkbox recipes should use those common tokens where possible, then map them into `--jb-checkbox-*` variables only where the checkbox needs a component-specific value.

## Styling Model

Use three layers when styling checkbox examples:

- Common design tokens are documented in [Theme Colors](../../jb-core/theme/stories/colors.mdx) and [Theme Sizes](../../jb-core/theme/stories/sizes.mdx).
- Reusable example theme packs live in [`docs/styles`](https://github.com/javadbat/design-system/tree/main/docs/styles) and can be shared across components.
- Component recipe files live in [`modules/jb-checkbox/stories/styles`](https://github.com/javadbat/design-system/tree/main/modules/jb-checkbox/stories/styles).
- Public component variables use the `--jb-checkbox-*` prefix and are the stable styling API.

Each example uses one shared example theme pack from [`docs/styles`](https://github.com/javadbat/design-system/tree/main/docs/styles) and one checkbox-specific recipe from [`modules/jb-checkbox/stories/styles`](https://github.com/javadbat/design-system/tree/main/modules/jb-checkbox/stories/styles). The style story applies both classes directly to `JBCheckbox`.

## Recommended Customization Path

1. Use `--jb-checkbox-*` variables first for component internals.
2. Use host styles for layout context around groups of checkboxes.
3. Use common theme variables from [Theme Colors](../../jb-core/theme/stories/colors.mdx) and [Theme Sizes](../../jb-core/theme/stories/sizes.mdx) when the style should align with the design system.
4. Put reusable theme-pack values in [`docs/styles`](https://github.com/javadbat/design-system/tree/main/docs/styles) only when they should be shared by multiple component demos.
5. Use `::part(...)` sparingly for official exposed surfaces such as `::part(check-bg)`, `::part(check-mark)`, `::part(label)`, and `::part(message)`.
6. Add a new component variable only when a needed internal part cannot be customized with the current public variables or official parts.

## Theme File Pattern

Use [Theme Colors](../../jb-core/theme/stories/colors.mdx) and [Theme Sizes](../../jb-core/theme/stories/sizes.mdx) as the common design-token guides. Theme-pack files should keep reusable demo variables in [`docs/styles`](https://github.com/javadbat/design-system/tree/main/docs/styles), then each component should keep its own style CSS in its story [`styles`](https://github.com/javadbat/design-system/tree/main/modules/jb-checkbox/stories/styles) folder.

```css
/* https://github.com/javadbat/design-system/blob/main/docs/styles/carbon.css */
.theme-name-style {
  /* shared design-system variables */
}

/* https://github.com/javadbat/design-system/blob/main/modules/jb-checkbox/stories/styles/style-carbon.css */
.theme-name-checkbox {
  /* checkbox-specific variables and official parts */
}
```

This keeps common theme tokens reusable and lets each component page show only the CSS it needs.
